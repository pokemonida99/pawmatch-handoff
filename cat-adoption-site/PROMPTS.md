# 認養網站生成 Prompt 組
### 由中途之家八輪研究彙整而成｜具 UX 設計思維與易用性規範
---
## 使用說明
- **Prompt A(主 Prompt)**:完整複製貼給 AI 網站生成工具(Claude Artifacts、v0、Lovable、Bolt、Cursor 皆可),一次生成整站 MVP。內容自足,不依賴其他上下文。
- **Prompt B1–B5(模組深化)**:主站生成後,逐一貼上以深化五個核心模組。
- **Prompt C(易用性驗收)**:每次生成後貼上,讓 AI 自我檢查並修正。
- 生成順序建議:A → 檢視 → C → B1~B5(依需要)→ C。
---
## Prompt A|主 Prompt(整站生成)
```
你是一位資深產品設計師兼前端工程師,請為「貓咪認養平台」建立一個單頁式響應式網站(React + Tailwind,行動優先)。這個平台的設計依據來自對台灣貓咪中途之家的深度研究,以下背景與原則是硬性規格,不是參考。
【研究背景(決定所有設計取捨)】
1. 認養人流失的主因是「被審查感」:傳統長問卷一放出去人就跑光;每家中途標準不一,認養人每換一家就要重新被審一次。
2. 退養率最低的是「共感型認養人」——先被貓的故事與個性打動的人;靠條件比對配上的人退養率高。
3. 中途最深的心理需求是「安心放手」而非「加快送出」:最終判斷權必須留在中途手上,系統只能當介紹人、不能當判官。
4. 送養後的追蹤(幸福回報)是留住成功認養的關鍵,但認養人討厭「被人檢查」;解法是去人化(系統中性提醒)+遊戲化(曬貓有舞台、回報可抽獎)。
5. 這個生態信任稀缺:金流與流程越透明,平台越可信。
【目標與成功指標】
主要目標:提高「開始申請→完成配對」的轉換率、降低退養率。
每一個設計決策若在「轉換率」與「被審查感」間衝突,一律優先消除被審查感。
【資訊架構(六個區塊,單頁滾動+錨點導覽)】
1. 首頁 Hero:一句以認養人利益出發的主張(例:「找到那隻跟你的生活合拍的貓」),主 CTA 是「開始配對」而非「瀏覽貓咪」;副 CTA「先逛逛貓咪」。禁止使用悲情救援影像,用溫暖日常的貓照。
2. 生活樣態配對測驗:對話式、一次只問一題、共 5–7 題(居住型態、在家時間、經驗、家中成員、期待的互動方式)。開場文案必須是利己框架:「回答幾個小問題,幫你找到合拍的貓」,絕不出現「審核」「資格」「申請條件」字眼。每題可跳過;進度以「還差 2 題就能看結果」的正向句式呈現。結果頁輸出 2–3 隻推薦貓+「為什麼合拍」的一句理由。
3. 貓咪列表與檔案頁:卡片以「個性敘事」為主視覺資訊——每隻貓一句第一人稱的個性介紹(例:「我是慢熟的觀察家,給我一週,我會睡在你鍵盤上」),品種、年齡、外觀條件次之。檔案頁結構:個性故事(最上)→ 生活需求(用貓的客觀需求陳述,如「我需要有人長時間在家」,禁止道德門檻句式如「你必須有愛心」)→ 健康護照(疫苗、驅蟲、結紮,時間軸呈現)→ 所屬中途介紹。
4. 認養旅程(儀式化四階段):以水平步驟條呈現「相識 → 見面 → 試相處 → 承諾日」,每階段說明認養人與中途各自要做什麼。文案基調是「迎接家人的旅程」,不是「通過審查的流程」。明確標示:最終決定由中途做出,並用一句話正面解釋為什麼(「每隻貓的中途最了解牠,他們的把關是給你和貓的保障」)。
5. 成長相簿(社會證明+回報系統預告):瀑布流展示已認養貓咪的近況照片牆(脫敏:只顯示貓名與月齡),每張卡片標「回家第 N 個月」。旁邊一小塊說明回報機制:「系統會溫柔提醒你曬貓,曬了還能抽獎」——用「曬貓」語言,禁用「回報」「追蹤」「檢查」。
6. 新手支援與透明區:三欄——新手指南(門窗防護、第一週相處、飲食入門)、匿名發問入口(強調「問題不是你的失敗,是每隻貓附贈的關卡」)、無責退回說明(「任何原因養不了,貓永遠可以回家,我們不追問」)+平台公開帳本入口(贊助與金流全公開)。
【UX 與易用性硬規則】
- 行動優先:所有互動元件觸控目標 ≥ 44px;單手可達的主 CTA 位置。
- 漸進揭露:任何表單首屏不超過 3 個欄位;配對測驗一屏一題。
- 認知負荷:每個區塊只有一個主要行動;全站主 CTA 顏色唯一。
- 可及性:語意化 HTML、圖片 alt、對比度 ≥ 4.5:1、可見的鍵盤 focus 樣式、尊重 prefers-reduced-motion。
- 錯誤與空狀態:說明發生什麼與怎麼做,不道歉不賣萌;空狀態是行動邀請。
- 微文案規範(全站一致):主動語態;按鈕寫明確動作(「看我的配對結果」而非「送出」);同一動作全程同名。全站禁用詞:審核、資格、申請條件、追蹤、檢查、監督、合格。替換詞:配對、認識、旅程、曬貓、近況。
- 信任訊號:中途實名與介紹、健康護照、透明帳本、無責退回——四者在頁面上都要出現至少一次。
【視覺方向】
避免 AI 生成網站的三種預設樣板(米白底+襯線大標+赤土色強調;近黑底+螢光單色;報紙式細線排版)。請依「貓的日常」自訂一套 4–6 色的溫暖但清爽的色票(可從貓毛色系取材:奶油、灰藍、暖橘、深炭),標題字與內文字刻意配對並說明理由;設計一個屬於本站的招牌元素(例如貓掌形的進度指示、或配對結果的「合拍度」以貓尾巴纏繞圖示呈現——擇一,全站只用一個記憶點)。整體氣質:像寵物友善咖啡店,不像政府網站也不像募款頁。
【交付】
單一檔案可運行的 React 元件,含假資料(至少 6 隻個性各異的貓、12 張成長相簿卡)。先用 3–5 句話說明你的設計計畫(色票、字體、招牌元素、取捨),再輸出程式碼。
```

## Prompt B1|配對測驗深化
```
深化「生活樣態配對測驗」模組:
1. 題目邏輯:居住(租/自有/與家人同住)→ 在家時間 → 養貓經驗 → 家中成員(幼兒/其他寵物)→ 期待互動(黏人/獨立/活潑/安靜)。租屋者追加一題「房東是否同意」,但語氣是提醒不是盤查:「先確認過房東,之後的旅程會順很多」。
2. 配對演算法:每隻貓有個性向量(活力、黏人度、獨處耐受、環境敏感、照護需求 1–5 分),與答案做加權距離,輸出前三名與「合拍理由」各一句(理由必須引用使用者的答案,例:「你說平日在家時間短——阿灰是獨處高手」)。
3. 沒有不及格:即使條件很受限,也輸出「目前最合拍」+誠實的一句提醒,並提供「先追蹤這隻貓的近況」的輕承諾選項,絕不顯示「無符合結果」。
4. 結果頁 CTA:「跟中途聊聊這隻貓」——開啟預填好的訊息草稿(含測驗摘要),寄給中途做最終判斷。摘要以「自我介紹」框架呈現給認養人確認後才送出。
```
## Prompt B2|貓咪檔案頁深化
```
深化貓咪檔案頁:
1. 個性故事區改為短篇敘事(120–180 字,第一人稱),敘事需包含:牠的來歷一句帶過(不賣慘)、兩個具體的行為細節、一個「適合什麼樣的人」的暗示。
2. 健康護照:垂直時間軸(救援日、疫苗×2、驅蟲、結紮、健檢),每項有日期與圖示;未完成項目顯示為「即將完成」而非紅色警告。
3. 「這隻貓的需求」用卡片呈現 3 項以內,全部以貓為主詞(「我需要窗戶有防護網」),每張卡附「怎麼做」的展開說明與採買參考。
4. 底部放該貓所屬中途的迷你檔案:照顧者暱稱、資歷、一句照顧哲學、其他待認養的貓。
```
## Prompt B3|成長相簿與回報系統
```
深化成長相簿模組為完整的「回家之後」體驗(前端示意即可):
1. 認養人視角:我的貓的時間軸(照片+月齡里程碑)、下次曬貓提醒的預告文案示例(中性、像日曆提醒:「小虎回家滿 3 個月了,牠的粉絲想看看牠」)、本月抽獎池展示(廠商贊助的罐頭/用品)。
2. 隱私控制永遠可見:每張照片可選「只給中途看/公開到相簿牆」,預設只給中途。
3. 求助入口與相簿併存:「遇到狀況?」按鈕開啟匿名發問,明確標示「發問不會影響任何紀錄」。
4. 無責退回的入口放在設定深處但可搜尋到,文案:「任何原因需要讓牠回家,都可以。我們只想確保牠安全。」——沒有原因欄位是必填。
```
## Prompt B4|中途後台(減負儀表板)
```
為中途角色生成一個簡潔後台(另一個路由或分頁):
1. 首屏是異常偵測儀表板:久未回報的認養(黃色關心,不是紅色警報)、待回覆的配對訊息、待更新的健康護照。
2. 每筆「久未回報」附一鍵操作:「送出溫柔提醒」(系統名義)或「私訊關心」(中途名義)——兩檔位設計,去掉催討的人、留下關係的人。
3. 配對訊息列表顯示測驗摘要卡,中途可標記「聊聊看/再想想/不合適」,不合適不通知對方原因、由系統以「這隻貓的中途覺得你們可能不是最合拍的,看看這幾隻?」轉介替代推薦。
4. 全後台語言以「減少你的工作」為框架,絕不出現績效、轉換率等字眼。
```
## Prompt B5|透明帳本頁
```
生成透明帳本頁:贊助收入與支出的公開流水(假資料),依「時報時銷」精神設計——每筆含日期、來源/用途、金額、憑證縮圖位。頂部一句定位:「每一塊錢的來去,都在這裡。」提供依月份與類別的篩選。視覺低調、像對帳單,不做成募款頁。
```

## Prompt C|易用性驗收(每輪生成後執行)
```
請以 UX 稽核員的身份,對剛生成的網站逐項自查並直接修正:
【轉換與框架】首屏 5 秒內能否理解「這裡幫我找到合拍的貓」?主 CTA 是否唯一且利己框架?全站搜尋一次禁用詞(審核/資格/追蹤/檢查/監督/合格),找到即改。
【流失點】配對測驗是否一屏一題、皆可跳過、進度正向?任何表單首屏是否 ≤3 欄位?
【信任】中途實名、健康護照、透明帳本、無責退回是否都至少出現一次?最終判斷權在中途,是否有正面解釋?
【易用性啟發式(Nielsen)】系統狀態可見、防錯與復原、一致性、辨識重於回憶、美學與極簡——各舉一處通過與一處待改並修正。
【可及性】鍵盤走完主流程、focus 可見、對比 ≥4.5:1、圖片 alt、reduced-motion。
【行動裝置】375px 寬度下無橫向捲動、觸控目標 ≥44px、主 CTA 拇指可達。
輸出:修正清單(問題→修法)+修正後的完整程式碼。
```

## 附:研究發現 → 網站功能對照表
| 研究發現 | 網站對應設計 | 對認養率的作用 |
|---|---|---|
| 長問卷造成前端流失 | 對話式測驗、一屏一題、可跳過 | 降低放棄率 |
| 共感型認養人退養最低 | 個性敘事優先的貓咪檔案 | 提高配對品質、降退養 |
| 審核=守門製造對立 | 配對取代資格、無不及格結果 | 消除被審查感 |
| 中途需保有最終判斷權 | 系統只做推薦與轉介、後台兩檔位提醒 | 中途願意進駐(供給面) |
| 回報的監視感 | 曬貓框架+抽獎+隱私預設私密 | 留任、社會證明反哺轉換 |
| 退養隱瞞比退養更糟 | 無責退回通道 | 守住貓的安全與品牌信任 |
| 信任稀缺的生態 | 透明帳本、健康護照、中途實名 | 建立平台公信力 |
